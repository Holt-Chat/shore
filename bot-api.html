The model

How a Holt bot works

A bot is not a webhook or a privileged server key. It is a real participant with the same shape as a person, owned and managed by a human.

A bot has its own username, display name, profile picture, and RSA keypair. A human account owns one or more bots and manages them with their normal logged-in session. Each bot authenticates with its own long-lived token, a 30-character nanoid, instead of the passkey challenge humans use to log in.

You generate the keypair client-side when you create the bot. The public key goes to the server as 392-character base64 DER; the private key stays with you and the server never sees it. The bot uses that private key to sign every message and to unwrap the per-channel symmetric keys, exactly like a human member. In encrypted channels the server holds only ciphertext.

A bot does not add itself. Someone already in the channel with the manage-members permission adds it by username, choosing the permissions it gets. A bot you mark public can also be added by anyone through an add-bot link; a private bot can only be added by its owner. Once inside, it reads and posts subject to the channel's permissions.

// owned

A human owns it

Create, edit, rotate, and delete bots from your own account. There is no separate bot console.

// keyed

It holds a keypair

The bot signs messages and unwraps channel keys with a private key you keep. The server only stores its public key.

// member

A manager adds it

A channel member who can manage members adds it by username, or anyone adds a public bot via its add-bot link. From there it is a normal end-to-end member.

Authentication

Two tokens, one header

Both schemes are HTTP bearer tokens. Which one you send depends on whether you are managing a bot as its owner or acting as the bot.

owner session

The human owner's session token. Authorizes the bot management endpoints under /bots and /bot/{bot_id}.

Authorization: Bearer <session_token>

bot token

The bot's own 30-character token. Authorizes the chat endpoints the bot calls at runtime: channels, messages, invites, and the stream.

Authorization: Bearer <bot_token>
The stream is the exception. EventSource can't set headers, so GET /stream takes auth as a query argument instead: ?authorization=Bearer <bot_token>. Same value, different place.
The token is shown once. Creating a bot and rotating a token both return the token a single time in the response. Store it then. It can't be read back later, only replaced by rotating it.

Slash commands

Let users invoke your bot by name

Register typed commands once; channel members trigger them by typing /command-name in the client. The app validates inputs against your schema before sending, so your bot receives a clean options map via SSE — no parsing, no freeform text matching.

Each command has a name (lowercase, 1–32 chars, [a-z0-9_-]), a description shown in the autocomplete menu, and an optional list of typed options. Options support six types: string, integer, number, boolean, user (a channel member ID), and channel (a channel the caller belongs to). Each option can carry constraints — min_length/max_length for strings, min_value/max_value for numbers, and a choices list of up to 25 fixed values for any type.

Commands are registered per-bot (not per-channel). Once a bot is a member of a channel, all its commands automatically appear in that channel's autocomplete. You can manage the full command set in one PUT call, or upsert and delete one at a time.

Receiving interactions

When a channel member triggers one of your commands, your bot receives an interaction_create event on the SSE stream. The server has already validated all option values against your schema — you get clean, typed data.

interaction_create SSE frame

event: interaction_create
data: {
  "id": "mZ7tLpV3nB1aWcQs0dRf",
  "channel_id": "aT4cR9mZ7tLpV3nB1xWd",
  "user": {
    "id": "9bZ2pQ7rLm4Xt1Vn3KdC",
    "username": "alice",
    "display": "Alice"
  },
  "command": "announce",
  "options": {
    "message": "Deployment complete",
    "priority": "urgent"
  }
}

Minimal Node.js handler

// listen on the SSE stream
const es = new EventSource(
  `${BASE}/stream?authorization=Bearer+${TOKEN}`
);
es.addEventListener("interaction_create", e => {
  const ix = JSON.parse(e.data);
  // ix.command, ix.options, ix.channel_id, ix.user
  if (ix.command === "announce") {
    postMessage(ix.channel_id, ix.options.message);
  }
});

Interactive components

Buttons and modals

Bots can attach clickable button rows to messages. When a user clicks a button, the bot receives a component_interaction SSE event and can respond with an acknowledgement or a modal form. Modal submissions arrive as modal_submit events.

Components work in all channel types. Bots use the same POST /channel/{id}/messages and PATCH /channel/{id}/message/{id} endpoints as users, with the addition of a components form field (JSON-encoded). Bots sign messages the same way users do — the same timestamp / signature fields apply.

post /channel/{channel_id}/messages bot token

Same endpoint as users. Bots sign messages the same way (multipart/form-data with content, timestamp, signature, and for encrypted channels key / iv). Bots additionally support a components field — a JSON-encoded array of up to 5 action rows.

es.addEventListener("component_interaction", e => {

Rich content

Embeds

Embeds are not a separate endpoint or field. They live inside the message content you already send, and every client renders them. A bot posts an embed by putting an ! embed: {json} line in the content of the normal send-message endpoint.

Each embed is one line that starts with ! embed: (optionally indented) followed by a JSON object. The object may span multiple lines; it is read by balanced braces. A single message may carry up to 10 embeds. Any text in content outside the embed lines is the normal message body.

In encrypted channels (type 1 and 2) the whole content, embeds included, is encrypted client-side like any other message. Put the ! embed: lines into the plaintext before you encrypt.

post embed fields

All fields are optional. Drop any you don't need. description and each field value render as markdown. url and author.url must be http or https; the renderer silently drops any other scheme. image, thumbnail, author.icon_url, and footer.icon_url are NOT plain URLs — clients refuse to auto-fetch an arbitrary http(s) image (that would leak the viewer's IP to whoever controls the URL just from opening the message). Instead, upload the image via POST /channel/{channel_id}/embed-asset below and put the returned {id, encrypted, iv} object directly in the field.

FieldTypeDescription
titlestringoptionalHeading shown at the top of the embed.
urlstringoptionalTurns the title into a link. http(s) only.
descriptionstringoptionalMain body text. Rendered as markdown.
colorint | stringoptionalLeft accent bar. An integer like 5793266 or a "#rrggbb" hex string.
authorobjectoptional{ "name", "url", "icon_url" }. Small line above the title; url links the name (http(s) only), icon_url shows a small icon (an embed-asset object).
fieldsobject[]optionalArray of { "name", "value", "inline" }. value is markdown; inline is a bool that lets fields sit side by side.
footerobjectoptional{ "text", "icon_url" }. Small line at the bottom; icon_url is an embed-asset object.
timestampint | stringoptionalUnix seconds or an ISO-8601 string. Shown in the footer.
imageobjectoptionalAn embed-asset object { "id", "encrypted", "iv" } from the upload endpoint below. Large image below the body.
thumbnailobjectoptionalAn embed-asset object { "id", "encrypted", "iv" } from the upload endpoint below. Small image in the top-right corner.

The body line and the embed line both go in the same content string, separated by a newline. The line above is the message body; the ! embed: line becomes the card.

post /channel/{channel_id}/embed-asset

Uploads a single image to reference from an embed's image, thumbnail, author.icon_url, or footer.icon_url field. Multipart form, one required field: image (the file, must be image/*). In encrypted channels (type 1 DM, type 2 group), encrypt the raw file bytes client-side with the channel's current AES key first — the same way message attachments are encrypted — and send encrypted=1, key (the current channel key id), and iv (16 chars) alongside it. In broadcast channels, just upload the raw image with no extra fields.

FieldTypeDescription
imagefilerequiredThe image file.
encryptedstringoptionalPass "1" to mark this upload as client-side encrypted (encrypted channels only).
keystringoptionalCurrent channel AES key id. Required when encrypted=1.
ivstringoptional16-char AES IV. Required when encrypted=1.

Responds with { "id", "encrypted", "iv", "success" } — put that object straight into the embed field. The upload is held in reserve (not yet visible to anyone) until you actually send a message whose content references its id; pass the file id(s) in a top-level embed_asset_ids JSON array field on the send-message request so the server knows to keep them. Uploads that never get referenced by a sent message are automatically cleaned up after about an hour.

Response

{
  "id": "V1StGXR8_Z5jdHi6B-myT",
  "encrypted": false,
  "iv": null,
  "success": true
}

Send the message with a matching embed_asset_ids: embed_asset_ids: '["V1StGXR8_Z5jdHi6B-myT"]' alongside content containing ! embed: {"image": {"id": "V1StGXR8_Z5jdHi6B-myT", "encrypted": false, "iv": null}}.

Rich content

Diagrams

Same mechanism as embeds: put a ! diagram: {json} line in content and every client renders it as an SVG diagram, drawn client-side from Mermaid syntax.

Rendering runs in Mermaid's strict security level, so HTML in node labels is not honored — this is a declarative diagram, not a place to run code. The resulting SVG is then loaded into a sandboxed <iframe sandbox> (no scripts, no same-origin, no network access at all) as a second layer of isolation. A single message may carry up to 10 embed/diagram/interactive blocks combined.

post diagram fields

One field.

FieldTypeDescription
codestringrequiredMermaid diagram source, up to 5000 characters.

Rich content

Interactive content

A ! interactive: {json} line lets you ship a small self-contained HTML/CSS/JS widget inside a message — same mechanism as embeds and diagrams, no separate endpoint.

Clients render it as a collapsed card behind a "Click to run" gate — it never executes automatically. When a viewer clicks it, the client builds an HTML document from your html/css/js and loads it into a sandboxed <iframe sandbox="allow-scripts"> (no allow-same-origin, so it can never read the viewer's Holt session, cookies, or DOM) with a strict Content-Security-Policy that blocks all network access — no fetch, no images/fonts over http(s), nothing leaves the frame. The card always shows a persistent notice that the content is not part of Holt Chat and hasn't been verified by it.

Because it still runs arbitrary script for whoever clicks it (just isolated and network-less), keep it to genuinely interactive things — small games, calculators, visualizations — not anything that needs to call out or that handles data you care about keeping private.

post interactive fields

All fields are optional, but you need at least one.

FieldTypeDescription
htmlstringoptionalMarkup for the document body, up to 20000 characters.
cssstringoptionalStylesheet, up to 10000 characters.
jsstringoptionalScript, up to 20000 characters. No network APIs will work — the frame's CSP blocks them.

Example content

Try the click counter:
! interactive: {"html": "<button id=b>Click me</button> <span id=n>0</span>", "js": "let n=0;b.onclick=()=>{n++;document.getElementById('n').textContent=n}"}

Errors & limits

One error shape, everywhere

Failed requests share a single JSON shape, and every endpoint is rate limited per IP and per identity.

On any failure, success is false and error is a human-readable message. Endpoints that can partially succeed, like editing a bot, also return an errors array listing the rejected fields.

Common statuses: 400 bad input, 401 missing or bad token, 403 not allowed, 404 not found, 413 file too large, 429 rate limited.

Rate limits return 429 with the X-RateLimit-Limit, X-RateLimit-Remaining, and X-RateLimit-Reset headers. The body's type tells you whether the per-IP or per-identity limit tripped.

Error body

{
  "error": "Channel not found",
  "success": false
}

Rate limit body · 429

{
  "success": false,
  "ratelimit": true,
  "type": "user"
}
